    • The absolute basics

    HTML (hyper–text markup language) is the standard way of structuring content on the web. It was proposed by Sir Tim Berners–Lee in 1989 while he was working as a contractor for CERN in Switzerland. In its earliest form, HTML was designed to be a system for researchers to use and share documents.

    It is a way of structuring words and pictures into blocks, so that software can better present it to the end user.

    The basic units of a HTML page, are called elements, tags or sometimes nodes. Each of these represents a block of content, like a paragraph of text or a table of results or an image.

    What are web standards?

    In the earlier versions of HTML (2.0 and 3.2), the browser vendors had complete control over what they chose to include, and how they did it. This meant that a default page with no styling would look completely different from browser to browser. In the early days, browsers even had their own proprietary tags, like Microsoft’s marquee or Netscape’s blink tag.

    The Web Standards Project began as a campaign to make building web sites more universal, by enforcing consistency on all browsers. The focus of this campaign was to both educate developers in how to follow the standards, as well as publically shaming the browser vendors who did not follow the standards themselves.

    A lot of the lasting antagonism aimed at Microsoft stems from this time. Microsoft decided, with the release of Internet Explorer 6, that browsers were done and they didn’t need to waste any further time on improving it. A browser called Firefox, an open source project built by a consortium of technology companies (including Microsoft, and later Apple and Google) soon proved that there was room for improvement. In later years, Apple began the Webkit project that now powers their browser Safari as well as Google’s Chrome, and the browser on most smartphones.

    Validators were used during this period to check that your code correctly followed the standards. Even now, some organisations are legally obliged to validate their code.

    Who sets the standards?

    An organisation made up of universities and technology companies, known as the World Wide Web Consortium (W3C). This body governs the standards that browser vendors follow, and define how the web works. A smaller group known as WHATWG (Web Hypertext Application Technology Working Group), with a stronger focus on web companies and the vendors themselves, decided that the W3C was moving too slowly and wrote their own HTML specification, this is what we know as HTML 5.

    The modern web uses parts of several different standards: XML and its parent SGML, XHTML 1.1, HTML 3.2, HTML 4.01 and finally HTML 5. A lot of the stricter rules and bad habits learned over time came from HTML 3.2 and XHTML 1.1.

    HTML 5?

    As a rapidly evolving specification, HTML 5 has many features that don’t even exist in browsers yet. It’s worth noting here, that when I mention HTML 5 I’m talking very specifically about the HTML part of the spec. There are many other areas now devoted to related and dependent web technologies.

    Comments

    Even though HTML is not exactly programming, every programmer is taught to write good documentation! The nature of HTML means that it does not need long explanations of why you have chosen one block over another, and since there is no real interactivity in HTML itself (coding logic should be in Javascript, CSS or a back–end script) comments are not strictly necessary. But they do exist:

    <!-- This is a HTML comment. -->

<![CDATA[
  This is an XML character data block, used to hide content from a validator. You may see Javascript blocks wrapped in these.
]]>

    Comments are often the poor man’s way of saving blocks of content for later, but remember that anyone can view the source of HTML and see it plainly. A better way of saving content you may want later is either copying it to a different file entirely, or using a proper version control system like Git.

    Attributes

    Each tag can, optionally, have attributes. These can be used to describe relationships, targets, identifiers or even alternate content.

    The one you will be most familiar with is the href of an a, the link that you are directing this anchor to go to. Also src on an img tells the browser where to look for the image file.

    <a href="http://google.com">
  <img src="/images/google.png" alt="Google">
</a>

    Another one you may have seen is the alt attribute of an img element (often referred to as an alt tag). This is an important accessibility feature that people with disabilities can use to describe what is in the image, if they’re not able to visually see it. In many large organisations it is a legal requirement to include appropriate alt attributes on images.

    Two further attributes that will become important when we discuss CSS later are class and id. As a general definition, class can be used on multiple elements while id should always refer to a single element only. More on this later.

    You may have noticed name in the form controls above, these are used by server–side scripts to recieve the user input information. Also, input has a few different types. Each label in the form has a for attribute pointing to its parent control’s id.

    Building a page

    A page needs, at minimum, a few elements to tell the browser what this document contains.

    <!DOCTYPE html>
<html>
<head>
  <meta charset="utf-8">
  <title>HTML/CSS Training</title>
  <meta name="description" content="Basic HTML and CSS training">
  <meta name="viewport" content="width=device-width">
  <link rel="stylesheet" href="app.css">
</head>
<body>

…

  <script src="//ajax.googleapis.com/ajax/libs/jquery/1.9.1/jquery.min.js"></script>
  <script src="app.js"></script>
</body>
</html>
    1. The first line sets the doctype of this document, an XML definition that tells the browser what type of document this is. In the past, these were tediously long and referenced a specification from the W3C. Now, all we do is say it’s HTML! All browsers that understand HTML 5 will recognise this.
    2. meta elements are used to describe features of the page, in this case that is uses utf–8 or Unicode as the character set.
    3. link is a special element used to relate another asset to this document, in this case the stylesheet in a file called app.css.
    4. If you are using Javascript on your page, the current best practice is to load it as the last element of your page. This will ensure that loading a large Javascript library will not cause a bottleneck in the page load. There is an exception to this rule! Typekit must always be called from the head section of a document, so that the font assets are downloaded before any content is shown. More on this later.

    The skeleton of elements shown above is a cut down version of the HTML 5 Boilerplate, an open source project to standardise how an empty page should be set up.

    Character entities

    HTML uses the ampersand character & as a control character, and validators will often complain about ampserands used without being escaped. An escaped ampersand looks like &amp;. Since HTML is built entirely on the use of < and >, these also need to be written as entities &lt; less than and &gt; greater than.

    Another character entity (sometimes referred to as an XML entity) that you may be familiar with is &nbsp; the non–breaking space. HTML ignores any whitespace after the first, so if you needed to have several spaces in a row you could use this entity. Don’t do this! This, along with overuse of br tags, is usually a sign that the CSS needs attention, spacing should never be defined in HTML.

    Do not put two spaces at the start of a sentence!

    Punctuation

    Punctuation can, and should, be done with character entities. There is an ongoing struggle by designers to educate the difference between a quote ’ and a prime '. Hint: primes are straight, quotes are curly!

    &ldquo;
    “ Left double quote
    &rdquo;
    ” Right double quote
    &lsquo;
    ‘ Left single quote
    &rsquo;
    ’ Right single quote or apostrophe
    &ndash;
    – En dash (half width)
    &mdash;
    — Em dash (full width)

    There are many useful characters that you can only cleanly use via character entities, such as accented characters for european languages, math symbols, asian characters and even some emoji are supported on certain operating systems (such as ☕). Any character in the Unicode table can be displayed in a HTML page. There are a number of ways to do this, but the easiest is by using the Hex code. eg. The coffee cup emoji is actually Unicode U+2615, we can enter this as &#x2615; on the page. (note: emoji is highly dependent on operating system fonts, so it pretty much only works on Macs!)

    Modern browsers have added a number of elements from HTML 5 to the set that we can use:

    header
    Header section
    section
    Structural section
    article
    Self–contained structural section
    footer
    Footer section
    nav
    Navigation section
    aside
    Breakout section
    figure
    Figure section
    figcaption
    Caption that goes in a figure

    In practice, most of these elements are simply pre–styled divs. A popular polyfill (a library to replicate new technology on old software) called html5shiv does exactly this for IE6-9, Safari 3 and Firefox 3 since they do not support these elements.

    Other elements have been reshuffled, given semantic meaning, or had semantic meaning taken away. Occasionally these changes are reversed a few months later. It’s not worth worrying about on a day–to– basis, but it is worth keeping an eye on it, so that you know you’re doing the best with what is available!

    Nested headings

    A cool new feature of HTML 5 is the ability to restart heading hierarchy within structural sections, ie.

    <section>
  <h1>Main Heading</h1>
  <article>
    <h1>Article Heading</h1>
    <aside>
      <h1>Heading</h1>
    </aside>
  </article>
  <article>
    <h1>Next Article Heading</h1>
    <aside>
      <h1>Heading for first aside</h1>
    </aside>
    <aside>
      <h1>Heading for second aside</h1>
    </aside>
  </article>
</section>

    In the above example, each article has its own h1 heading, and each aside within it has its own hierarchy of headings as well. Before HTML 5, there was only one hierarchy of headings, now they are all encapsulated in their own section.

    Form controls

    Another major addition in HTML 5 is a larger set of form controls, almost matching what can be done with desktop software. Most of these are not well supported at the moment, but it’s worth looking at demonstrations on the web of what we’ll be seeing soon. There are some features that have a little broader support, such as placeholder attributes and the search input type, but I would be careful about using these unless you know the audience for your site is only using modern or mobile browsers.

    • How to CSS

    CSS (Cascading Style Sheets) are the visual styles applied to the structured content of your HTML. Styles are defined by setting properties for each element, and properties may be inherited through the structure set in your HTML.

    The easiest way to explain this will be to look at the stylesheet for this page!

    Notes on CSS

    • @import is the CSS way of including another file, this is the equivalent link in HTML. These two imports are including a global normaliser and a webfont library, these will both be discussed below.
    • Properties are separated by a semi–colon ; and wrapped with curly braces { and }.
    • Definitions that begin with a . are referring to any HTML element with a class by that name.
    • Definitions that begin with a # are referring to the unique HTML element with that id.
    • Pseudo–selectors begin with a colon : and are specific to certain HTML elements. a has :hover, for example, a mutation effect that allows you to style when the user’s mouse hovers over that element. :hover is the entire basis of CSS rollover menus.
    • Comments begin with /* and end */. Note that, by convention, I list the font stacks and colours used (ie. this is not required).
    • Like any code, there are numerous ways to format CSS — single property per line or stacked, indented or left aligned, full property or minimised. The coding style is purely choice, it has little if any impact. I have my own coding style developed over many years of doing this, but very few web professionals agree on this!
    • Units can be expressed in px for pixels, em for ems (a typographic unit, the width of the capital M in the selected typeface) or % for percent of an element’s parent container.
    • Colours can be expressed in hex, eg. #fff or rgb (red, green, blue) values eg. rgb(255,255,255). There is also a variant of rgb called rgba, where the a is an alpha (transparency) channel eg. rgba(255,255,255,0.4). This definition would be the same colour as the other two, but only at 40% opacity.
    • Properties can be stacked, eg. margin:0; is the same as margin:0 0 0 0; (top right bottom left). There are also the more complicated stacks such as two–point margin:0 1em; which is the same as margin:0 1em 0 1em; (bottom and left are inherited from top and right) and three–point margin:1em 1em 0; which is the same as margin:1em 1em 0 1em; (it inherits the left from right). Complicated!
    • Some properties can be grouped or minimised, eg. border:1px solid #000; is the same as border-width:1px; border-style:solid; border-color:#000; (in fact, the border property is a helper for setting all four groups of border properties at once, so it is actually equivalent to border-left-width:1px; border-left-style:solid; border-left-color:#000; border-right-width:1px; border-right-style:solid; border-right-color:#000; border-top-width:1px; border-top-style:solid; border-top-color:#000; border-bottom-width:1px; border-bottom-style:solid; border-bottom-color:#000; — this is why I call it a helper)
    • Font can by minimised as font:italic 13px/1.2 "Source Sans Pro", Arial, Helvetica, sans-serif; but I would recommend against doing so. Some properties are better to define separately, and parts of this particular example will change often enough to warrant it. Font base size should be set in pixels once at a high level (eg. body), and manipulated relatively with ems whenever you can.
    • line-height can take units, but works a lot better without. Unitless line-height properties work as a ratio of the computed font size (ie. line-height:1.2; will calculate the line height as 120% of the font size).

    Rather than list all the properties here, I’ll instead link to this definitive list of properties published by Mozilla.

    Resets

    In the early days of the web when browsers had their own set of default properties on elements, a stylesheet known as a global reset was needed to set everything in a browser back to a consistent default. This allowed developers to make a single stylesheet that could be used across all browsers, without needing to tune things separately for each.

    Browsers are more uniform now, but still have significant differences in their default styling. A more modern approach to resetting style is known as normalize (as with the property "color", most terms are defined by Americans and have their spelling). Unlike the reset which took everything back to 0, normalize sets reasonable defaults instead, providing a safe starting point to apply your own styles.

    Webfonts

    The options for changing typography on the web were not great until Javascript–served solutions (like Typekit) were launched. CSS has always had a font-face property that allows the web developer to reference a font asset, but licensing expensive font assets from the foundries had been difficult until then. Techniques like SIFR (a flash–embedded solution) and Cufon (a Javascript translation of the font asset), as well as rendering the font out as an image were all in use, but none of them were ideal. In general, there was a bare minimum set of fonts known as the "Web Fonts" that you could expect all browsers to be able to use. This included Arial, Georgia, Times New Roman, Century Gothic, Courier New, Trebuchet MS, Tahoma and Verdana (fonts that came pre–installed on Windows).

    There are a few good services in the space now, including:

    • Typekit the major innovator, and the most popular service now owned by Adobe (arguably the biggest font foundry)
    • Adobe Edge Web Fonts a service that uses the same technology as Typekit to serve free fonts
    • Google Fonts a similar service that only serves free fonts, some of which are quite good
    • Font Squirrel has an online tool to generate the necessary assets to display on all browsers, but beware that using this may not be is rarely covered in the license of the font you wish to use

    Symbol fonts

    Once again, the techniques and ideas have been around for a long time, but the inclusion of an open source icon project known as Fontawesome in Twitter Bootstrap (a design framework) has popularised the concept of symbol fonts. A symbol font is made up of vectors of icons instead of characters, and can be used to display crisp icons on high resolution displays (like Apple’s various retina devices or the recent Android phones).

    Another alternative to this is SVG (scalable vector graphics), an older derivative of XML that uses a code structure rather than binary data to describe a vector graphic. A good resource for SVG icons is The Noun Project.

    Vendor prefixes

    When experimenting with new technologies, browser vendors will often add their prefix to a property until they are happy with the performance and/or the standard is solid. An example of this is border–radius, a property to round the corners of an element. This has had such strong support from browser makers that all modern browsers now support it, but initially they were tested with -moz-border-radius for Firefox and -webkit-border-radius for Safari (and later Chrome). This practice still continues today, and it means that if you want to use a new technique, you may need to set it against a number of different vendors. Current best practice is to list all vendor prefixed properties, then an unprefixed one for future proofing the eventual support. eg.

    p {
  -moz-super-property:6px rgba(0,0,0,0.3);
  -webkit-super-property:6px rgba(0,0,0,0.3);
  -ms-super-property:6px rgba(0,0,0,0.3);
  super-property:6px rgba(0,0,0,0.3);
}

    Where -moz targets Firefox, -webkit targets Safari, Chrome and mobile browsers and -ms targets Internet Explorer version 9+. Google recently announced that they are formally moving Chrome away from webkit, but they do not intend to use a vendor prefix with their new engine.

    • Markdown

    Markdown is a newer markup language created by John Gruber, and later Aaron Swartz. The purpose of markdown is allowing writers to write in a simplified text format, and then convert this into HTML. Markdown is becoming popular as a way of quickly structuring content, especially in a CMS.

    Markdown looks like this:

    # Heading 1

Paragraph is defined by line space.

This second paragraph has an *italic* word and a **bold** one, for example.

[A link to a great markdown resource!](http://howtomarkdown.com)

    This would render to:

    <h1>Heading 1</h1>
<p>Paragraph is defined by line space.</p>
<p>This second paragraph has an <em>italic</em> word and a <strong>bold</strong> one, for example.</p>
<p><a href="http://howtomarkdown.com>A link to a great markdown resource!</a></p>
